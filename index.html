Business Context & Rationale
Comprehensive justification for the Greenfield Local Hub platform, KPIs, UAC and legislation
Cooperative Business Rationale

Greenfield Local Hub is a digital cooperative platform modelled on the principles of organisations such as Arla Foods, where producers collectively own and govern the distribution of their output. In a traditional supply chain, intermediaries (supermarkets, wholesalers) capture the majority of the margin, leaving local farmers and producers with little commercial control or consumer visibility. This platform addresses that imbalance directly by providing a direct-to-community digital channel that eliminates intermediary markups and passes value back to both producers and consumers.

From a sustainability perspective, short supply chains substantially reduce food miles — the distance food travels from production to consumer — directly lowering the cooperative's carbon footprint and contributing to the UK Government's Net Zero 2050 target. By surfacing producer profiles and farm stories alongside products, the platform also builds consumer trust and brand loyalty, which is evidenced in consumer research as a primary driver of purchasing decisions in the local food sector.

The loyalty scheme is specifically designed to incentivise repeat purchasing behaviour through a progressive discount model (5% at £25, 10% at £75, 20% at £150 cumulative spend). This increases customer lifetime value (CLV) and reduces churn — two core metrics for cooperative viability. Meanwhile, the admin stock tracing module (linking every product to a batch, a producer, and a delivery date) fulfils food safety traceability obligations under the UK Food Safety Act 1990 and EU Regulation (EC) 178/2002, which require food businesses to be able to trace products "one step forward, one step back" in the supply chain.

Requirements Analysis — FURPS Model
Comprehensive functional and non-functional requirements using the FURPS framework
FURPS stands for Functionality, Usability, Reliability, Performance, Supportability. It is an industry-standard model for capturing both what a system does (functional) and how well it does it (non-functional). Using FURPS demonstrates a thorough understanding of the business context required for a Distinction.

Loyalty Scheme — Full IPO Decomposition

Distinction requires comprehensive decomposition of complex processes into distinct Inputs, Processes, and Outputs. The loyalty discount calculation is the most algorithmically significant process in the system.

📥 INPUTS

  • → userID (INT) — identifies the customer
  • → basketTotal (DECIMAL) — current order value before discount
  • → cumulative_spend (DECIMAL) — fetched from DB: total historically spent by this user
  • → Loyalty tier constants: £25, £75, £150
  • → Discount rate constants: 5%, 10%, 20%

⚙️ PROCESSES

  • 1. Query DB for user's cumulative_spend
  • 2. Compare spend against tier thresholds (nested IF)
  • 3. Assign discountRate based on highest qualifying tier
  • 4. Calculate: discountAmount = basketTotal × discountRate
  • 5. Calculate: finalTotal = basketTotal − discountAmount
  • 6. Round finalTotal to 2 decimal places (currency)
  • 7. On order success: increment cumulative_spend by basketTotal
  • 8. Update loyalty_tier field in users table

📤 OUTPUTS

  • ← tierLabel (STRING) — displayed to user
  • ← discountRate (DECIMAL) — e.g. 0.05
  • ← discountAmount (DECIMAL) — e.g. £0.93
  • ← finalTotal (DECIMAL) — charged at checkout
  • ← Updated cumulative_spend in DB
  • ← Updated loyalty_tier in DB
  • ← discount_applied stored in orders table

Algorithm 2 — Loyalty Discount Calculation

SEQUENCE · SELECTION
Pseudocode
// Algorithm: Calculate Loyalty Discount // Inputs: userID, basketTotal // Outputs: discountRate, finalTotal CONSTANT TIER_1_THRESHOLD ← 25 CONSTANT TIER_2_THRESHOLD ← 75 CONSTANT TIER_3_THRESHOLD ← 150 CONSTANT TIER_1_RATE ← 0.05 CONSTANT TIER_2_RATE ← 0.10 CONSTANT TIER_3_RATE ← 0.20 BEGIN CalculateDiscount(userID, basketTotal) // Fetch user's cumulative spend from DB totalSpend ← DB_QUERY( "SELECT cumulative_spend FROM users WHERE id = ?", [userID] ) // Determine discount tier IF totalSpend >= TIER_3_THRESHOLD THEN discountRate ← TIER_3_RATE tierLabel ← "Gold — 20% OFF" ELSE IF totalSpend >= TIER_2_THRESHOLD THEN discountRate ← TIER_2_RATE tierLabel ← "Silver — 10% OFF" ELSE IF totalSpend >= TIER_1_THRESHOLD THEN discountRate ← TIER_1_RATE tierLabel ← "Bronze — 5% OFF" ELSE discountRate ← 0 tierLabel ← "No discount yet" END IF // Apply discount to basket discountAmount ← basketTotal * discountRate finalTotal ← basketTotal - discountAmount finalTotal ← ROUND(finalTotal, 2) OUTPUT tierLabel, discountAmount, finalTotal RETURN finalTotal END CalculateDiscount

Test Strategy
Testing approach, component order, test types and initial test plan for Greenfield Local Hub

⬜ White Box Testing

Developer-led · Internal Logic
Purpose
Tests the internal code structure. Every conditional branch, loop and function is tested to ensure the code itself works correctly (not just inputs/outputs). Developer tests their own code during development.
Who Performs
Developer (student)
Test Data
All possible input combinations, including boundary values. Each path through every conditional (IF/ELSE, WHILE, FOR) exercised at least once.
Test Criteria
Output matches programmer's intended logic for every input. All branches covered.
When
Throughout development — after each module/function is written, before integration.
Est. Time
~30 min per module; full white-box pass ~2–3 hours total.

⬛ Black Box Testing

Input/Output based · User perspective
Purpose
Tests system behaviour from the user's perspective without knowledge of internal code. Verifies that given an input, the correct output/behaviour is produced as defined in the requirements.
Who Performs
Developer and selected test users (residents, community members)
Test Data
Valid, valid extreme, invalid, invalid extreme, and erroneous data for all user inputs.
Test Criteria
System behaves as per specification. Correct success/error messages shown. No data corruption.
When
After each feature is complete and after full system integration.
Est. Time
~1 hour per major feature; full black-box pass ~4–5 hours.

Security Controls & CIA Framework
Exact PHP/SQL procedures ensuring Confidentiality, Integrity, and Availability
CIA Triad: Every security measure below is mapped to one or more of the three core security aims — Confidentiality (data is only accessible to authorised users), Integrity (data is accurate and untampered), and Availability (the system remains accessible to legitimate users).
1 — Password Hashing (Confidentiality)

Passwords must never be stored as plain text. PHP's password_hash() uses the bcrypt algorithm, which applies a salt and a cost factor making brute-force attacks computationally expensive.

// REGISTRATION: Hash password before storing $password = $_POST['password']; $hash = password_hash($password, PASSWORD_BCRYPT, [ 'cost' => 12 // Higher = slower = more secure ]); // Store $hash in DB, never $password // LOGIN: Verify against stored hash if (password_verify($inputPassword, $storedHash)) { // Grant access } else { // Increment failed attempts }
Confidentiality
2 — Prepared Statements (Integrity)

SQL Injection is one of the most dangerous vulnerabilities (OWASP Top 10). Prepared statements separate SQL logic from user data — the DB driver handles escaping, so malicious input like admin' OR '1'='1 is treated as literal data, not SQL code.

// UNSAFE — never do this: $sql = "SELECT * FROM users WHERE email='$email'"; // SAFE — prepared statement with PDO: $stmt = $pdo->prepare( "SELECT id, password_hash, role FROM users WHERE email = ? AND is_active = 1" ); $stmt->execute([$email]); $user = $stmt->fetch(PDO::FETCH_ASSOC);
Integrity
3 — Input Validation & XSS Prevention (Integrity)

Cross-Site Scripting (XSS) allows attackers to inject scripts into pages viewed by other users. All output must be sanitised. User-submitted content (noticeboard posts, forum comments) is especially vulnerable.

// Validate email format server-side $email = filter_var($_POST['email'], FILTER_VALIDATE_EMAIL); if (!$email) { die("Invalid email"); } // Strip HTML tags from text inputs $name = strip_tags(trim($_POST['name'])); // Encode output before displaying to browser echo htmlspecialchars($userInput, ENT_QUOTES, 'UTF-8'); // Converts < > " ' & to HTML entities
Integrity
4 — CSRF Protection (Integrity)

Cross-Site Request Forgery tricks authenticated users into unknowingly submitting malicious requests. A unique CSRF token is generated per session and validated on every state-changing form submission (login, order, post).

// Generate token on page load if (empty($_SESSION['csrf_token'])) { $_SESSION['csrf_token'] = bin2hex(random_bytes(32)); } // Embed in form as hidden field <input type="hidden" name="csrf_token" value="<?= $_SESSION['csrf_token'] ?>"> // Validate on form submission if (!hash_equals($_SESSION['csrf_token'], $_POST['csrf_token'])) { die("CSRF validation failed"); }
Integrity
5 — Role-Based Access Control (Confidentiality)

Every admin page must verify the user's role on every page load — not just at login. A user who has their session cookie stolen must not be able to access admin features.

// auth_check.php — included at top of every admin page session_start(); if (!isset($_SESSION['user_id']) || $_SESSION['role'] !== 'admin') { header('Location: /login.php?error=unauthorised'); exit; // exit() CRITICAL — stops page rendering } // Session timeout — auto-logout after 30 mins idle if (isset($_SESSION['last_active']) && (time() - $_SESSION['last_active'] > 1800)) { session_destroy(); header('Location: /login.php?error=timeout'); exit; } $_SESSION['last_active'] = time();
Confidentiality
6 — Brute Force & DoS Protection (Availability)

Without rate limiting, attackers can attempt thousands of password combinations per second. A lockout system combined with HTTPS and secure headers protects availability and confidentiality.

// Check failed attempts before processing login $stmt = $pdo->prepare( "SELECT failed_attempts, locked_until FROM users WHERE email = ?" ); $stmt->execute([$email]); $row = $stmt->fetch(); if ($row['locked_until'] > date('Y-m-d H:i:s')) { die("Account locked. Try again later."); } if ($row['failed_attempts'] >= 5) { // Lock account for 15 minutes $unlock = date('Y-m-d H:i:s', strtotime('+15 min')); $pdo->prepare("UPDATE users SET locked_until=? WHERE email=?") ->execute([$unlock, $email]); }
AvailabilityConfidentiality
7 — HTTPS & Secure HTTP Headers

HTTPS encrypts all data in transit using TLS, preventing man-in-the-middle attacks. Secure headers add an additional layer of protection in the browser.
